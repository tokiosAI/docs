Tokios lets you expose multiple models — each under its own public name — and route between them by changing the `model` field in your request. The endpoint and API key stay the same; only the `model` value changes.

## How routing works

Each deployment has a unique public name. When Tokios receives a request, it reads the `model` field, finds the deployment with that name, rewrites it to that deployment's upstream model id, and forwards it down the connector's tunnel. Your client config never changes — only the `model` value.

## One connector, many models

A single connector can serve several models. List each upstream model id in `Routes` and map it to a named local upstream in `Upstreams`:

```json tokios-connector.json
{
  "Connector": {
    "ConnectorId": "con_...",
    "Token": "ct-...",
    "Routes": [
      { "Model": "gemma4:e4b",    "UpstreamId": "ollama" },
      { "Model": "qwen2.5-coder", "UpstreamId": "lmstudio" }
    ],
    "Upstreams": {
      "ollama":   { "BaseUrl": "http://127.0.0.1:11434/v1" },
      "lmstudio": { "BaseUrl": "http://127.0.0.1:1234/v1" }
    }
  }
}
```

Then register a **deployment** for each model in the console, giving each a public name (e.g. `gemma-tunnel` → `gemma4:e4b`, `qwen-coder` → `qwen2.5-coder`), all pointing at this connector.

<Tip>
  Prefer to keep them separate? You can also run one connector per model, each with its own `ConnectorId`/`Token` and `Upstream`. Either approach works — the server only ever sends the model id down the tunnel.
</Tip>

## Switching models in requests

With both deployments registered, switch models by changing only the `model` field:

```bash
# Use the Gemma model
curl https://api.tokios.com/v1/chat/completions \
  -H "Authorization: Bearer sk-tok-YOUR_KEY" \
  -H "Content-Type: application/json" \
  -d '{"model": "gemma-tunnel", "messages": [{"role": "user", "content": "Hello"}]}'

# Use the Qwen model
curl https://api.tokios.com/v1/chat/completions \
  -H "Authorization: Bearer sk-tok-YOUR_KEY" \
  -H "Content-Type: application/json" \
  -d '{"model": "qwen-coder", "messages": [{"role": "user", "content": "Hello"}]}'
```

The endpoint, the auth header, and every other parameter stay identical. Tokios handles the dispatch.

## Swap the backend without touching clients

Because clients only reference the public name, you can move a model to different hardware or a different backend without changing any client config: update the deployment (or the connector's upstream) to point at the new server, keep the **same public name**, and existing requests keep working.
